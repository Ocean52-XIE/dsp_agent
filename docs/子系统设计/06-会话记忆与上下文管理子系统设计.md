# 会话记忆与上下文管理子系统设计

## 1. 目标
为多轮问答和问题分析提供稳定的会话状态管理，避免上下文丢失、重复检索和推理漂移，并为 LangGraph 的中断恢复提供持久化基础。

## 2. 记忆分层

### 2.1 短期记忆

保存当前会话最近若干轮的原始消息，用于直接上下文理解。

### 2.2 摘要记忆

当会话变长时，提炼出结构化摘要，例如：

- 当前主题
- 当前任务阶段
- 已确认模块
- 已知症状
- 已引用关键证据
- 尚未回答的问题

### 2.3 工件记忆

保存分析链中的中间产物：

- 检索结果
- 候选模块
- 根因假设
- 修复方案
- 用户确认状态
- 最近一次可复用的问题分析结果

## 3. 状态存储设计

| 存储对象 | 建议介质 | 用途 |
| --- | --- | --- |
| 会话主表 | PostgreSQL | 会话元信息 |
| 消息明细 | PostgreSQL | 用户/助手消息 |
| Graph 状态快照 | PostgreSQL / Redis | 中断恢复 |
| 热点缓存 | Redis | 加速最近会话读取 |

## 4. 上下文裁剪策略

为控制 token 成本，建议按优先级拼接上下文：

1. 当前轮用户输入
2. 最近 3 到 5 轮强相关消息
3. 结构化摘要
4. 当前任务的关键证据
5. 需要恢复的中间状态

其中“需要恢复的中间状态”至少应包括：

- `active_task_stage`
- `active_topic`
- `active_module_name`
- `last_analysis_result`
- `pending_action`

低相关历史内容不直接注入模型。

## 5. 会话标识设计

- `session_id`：整个会话唯一标识
- `message_id`：单条消息标识
- `trace_id`：单轮执行链路标识
- `checkpoint_id`：Graph 中断恢复点标识

## 6. 中断恢复机制

在以下节点必须落快照：

- 路由完成后
- 检索完成后
- 问题分析完成后
- 等待代码确认前

恢复时根据 `checkpoint_id` 直接装载状态，避免整轮重跑。

## 7. 多轮追问设计

追问时需要区分两种场景：

- `补充信息`：用户在完善当前问题，不应开启新线程
- `新问题`：应新建会话或提示切换主题

可通过主题相似度、当前任务状态和代词指代判断。

典型规则包括：

- 若当前轮出现“它、这个、这块”等代词，优先尝试复用 `active_topic`
- 若当前轮从 `knowledge_qa` 切换到 `issue_analysis`，视为同主题阶段升级
- 若当前轮直接提出“给我代码实现”，且存在 `last_analysis_result`，视为从分析升级到代码实现
- 若模块、主题和阶段都不连续，则视为新任务

## 8. 清理与归档

- 长时间未访问的会话可转冷存储
- 对高价值会话可沉淀为新的案例数据
- 过期调试快照应定期清理

## 9. 风险与应对

| 风险 | 表现 | 对策 |
| --- | --- | --- |
| 上下文过长 | 成本和延迟上升 | 摘要化与分层裁剪 |
| 恢复状态不一致 | 图执行异常 | 关键节点落快照并带版本号 |
| 主题串线 | 回答偏题 | 新问题检测与线程切换提示 |

## 10. 验收标准

- 支持多轮连续对话
- 支持确认节点后的恢复执行
- 支持长会话摘要与上下文裁剪
- 能保留关键中间工件供前端与调试使用
