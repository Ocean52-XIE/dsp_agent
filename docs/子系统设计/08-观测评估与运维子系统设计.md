# 观测评估与运维子系统设计

## 1. 目标
为系统提供端到端可观测能力、质量评估能力和运维手段，确保智能问答与问题分析链路可追踪、可评估、可持续优化。

## 2. 可观测性范围

- API 请求日志
- LangGraph 节点执行链路
- 检索召回情况
- 模型调用耗时与成本
- 用户反馈与采纳结果

## 3. 日志设计

### 3.1 关键日志字段

- `trace_id`
- `session_id`
- `message_id`
- `user_id`
- `intent`
- `node_name`
- `latency_ms`
- `model_name`
- `prompt_tokens`
- `completion_tokens`
- `retrieval_count`
- `error_code`

### 3.2 日志分层

- `access log`：请求入口日志
- `graph log`：节点运行日志
- `retrieval log`：召回与重排日志
- `feedback log`：用户反馈、人工修正日志

## 4. Tracing 设计

建议用 `OpenTelemetry` 打通以下 Span：

- `http.request`
- `graph.domain_gate`
- `graph.intent_classifier`
- `retrieval.wiki`
- `retrieval.case`
- `retrieval.code`
- `analysis.root_cause`
- `generation.code`

这样可定位慢点、失败点和高成本节点。

## 5. 指标体系

### 5.1 在线指标

- 请求 QPS
- 成功率
- 平均响应时长
- 流式首字节时延
- 模型调用错误率
- 检索超时率

### 5.2 业务质量指标

- 路由准确率
- 拒答准确率
- 问答满意度
- 模块定位准确率
- 修复建议采纳率
- 代码建议使用率

## 6. 评估体系

### 6.1 离线评测

建立标准数据集：

- 知识问答样本
- 问题分析样本
- 无关问题样本
- 代码定位样本

离线跑批后输出：

- 正确率
- 引用覆盖率
- 幻觉率
- 平均成本与耗时

### 6.2 在线反馈

前端提供：

- 有用/无用反馈
- 结论是否采纳
- 哪一部分不准确

这些反馈回流到样本池，用于检索和 Prompt 优化。

## 7. 运维能力

- 支持按 `trace_id` 查看整轮执行详情
- 支持查看命中证据与节点耗时
- 支持重放某次请求做回归分析
- 支持索引构建任务监控与告警

## 8. 告警策略

对以下情况设置告警：

- API 错误率超过阈值
- 路由准确率异常波动
- 检索命中数大幅下降
- 模型平均延迟飙升
- 索引任务连续失败

## 9. 风险与应对

| 风险 | 表现 | 对策 |
| --- | --- | --- |
| 只看系统指标，不看业务质量 | 线上看似正常但答案不好用 | 建立业务质量看板 |
| 日志过多导致成本高 | 存储压力大 | 对长文本做摘要存储 |
| 无法复盘错误回答 | 问题难定位 | 保留 trace、证据、状态快照 |

## 10. 验收标准

- 能通过 trace_id 追踪整轮链路
- 能统计核心性能指标和业务质量指标
- 能接收用户反馈并沉淀为优化输入
- 能对索引和模型链路异常及时告警
